# Code Review — 2026-07-02 (Claude Fable 5)

Review of `jsonstrict` at commit `f2c2cec`. All bugs below were verified
empirically against a local build, comparing behavior with `encoding/json`.

## Bugs

- [x] **Stack overflow on self-referential embedded types** — `knownJSONKeys`
  (unmarshal.go:98-108) recurses into embedded structs with no cycle
  detection. A legal type like `type Node struct { *Node; X int }` crashes
  with `fatal error: stack overflow`. `encoding/json` handles the same type
  fine. Fix: thread a visited-type set through the recursion.
  - [x] Add regression test with a recursive pointer-embedded type.
    (`TestUnmarshal_RecursiveEmbedded`, `TestUnmarshal_MutuallyRecursiveEmbedded`)

- [x] **Embedded structs with a JSON tag are wrongly flattened** —
  unmarshal.go:98 checks `field.Anonymous` before reading the tag, but
  `encoding/json` treats a *tagged* embedded field as a regular named field.
  For `Inner` embedded with tag `json:"inner"` and input
  `{"inner":{"a":"x"},"b":"y"}` (which stdlib decodes cleanly), jsonstrict
  reports `inner` as unknown **and** `a` as missing — both false. Fix:
  consult the tag first; only untagged embedded structs flatten.
  - [x] Add test: tagged embedded struct produces no false diagnostics.
    (`TestUnmarshal_TaggedEmbeddedIsNamedField`, `TestUnmarshal_TaggedEmbeddedNotFlattened`)

- [x] **`json:"-"` on an embedded struct is ignored** — same root cause: the
  `-` check (unmarshal.go:116) is never reached for anonymous fields.
  stdlib excludes the embedded struct entirely; jsonstrict recurses in and
  treats its fields as known/required, silently passing inputs stdlib's
  strict decoder rejects. Fix falls out of the tagged-embed fix above.
  - [x] Add test: `json:"-"` embedded struct's fields are reported unknown.
    (`TestUnmarshal_DashEmbeddedExcluded`)

- [x] **Misleading error for non-struct pointers** —
  `Unmarshal(data, new(int))` returns `*json.InvalidUnmarshalError`, which
  renders as `json: Unmarshal(nil *int)` even though the pointer is not nil
  (unmarshal.go:57-60). That error type only knows stdlib's constraints, not
  this package's "must point to a struct" rule. Fix: define a dedicated
  error type (keep `*json.InvalidUnmarshalError` for the nil/non-pointer
  cases stdlib itself covers). Fixed as `InvalidTargetError`.
  - [x] Update `TestUnmarshal_NonStructPointerReturnsError` for the new type.
  - [x] Document the error contract in the `Unmarshal` doc comment.

## Precision gaps

- [x] **`omitzero` not recognized** — Go 1.24's `encoding/json` added
  `omitzero`, semantically optional like `omitempty`, but unmarshal.go:123
  only checks `omitempty`, so `json:"f,omitzero"` fields are reported
  missing. Fix: treat `omitzero` as optional too. Option matching is now
  whole-option (`tagOptContains`), like stdlib's `tagOptions.Contains`.
  - [x] Add test for `omitzero` fields not reported missing.
    (`TestUnmarshal_OmitzeroNotMissing`)

- [x] **Top-level `null` behavior undocumented** —
  `Unmarshal([]byte("null"), &v)` returns nil error, an untouched struct,
  and every required field as missing. Defensible but surprising.
  - [x] Document the behavior in the package/function doc comment.
  - [x] Add a test locking it in. (`TestUnmarshal_TopLevelNull`)

- [x] **`{"key":null}` counts as present, not missing** — undocumented, and
  present-but-null vs. absent is exactly the distinction callers of this
  package care about.
  - [x] Document explicitly. (Unmarshal doc comment and README)
  - [x] Add a test locking it in. (`TestUnmarshal_NullValueIsPresent`)

- [x] **Shadowed field names can land in both required and optional** — an
  outer field and an embedded field sharing a JSON name are both merged via
  `maps.Copy` (unmarshal.go:105-106); a key can be reported missing even
  though stdlib resolves the conflict to the shallower, optional field. The
  doc comment covers same-depth conflicts but not this shallower-wins case.
  - [x] Resolve shallow-wins shadowing (or document the limitation).
    `knownJSONKeys` now returns per-field info (optional/tagged/depth) and
    resolves conflicts like stdlib: shallower wins, tagged beats untagged at
    equal depth; remaining same-depth ties stay a documented limitation.
    (`TestUnmarshal_ShadowedFieldShallowerWins`)

- [x] **Tag names not validated** — stdlib rejects invalid tag names
  (`isValidTag`) and falls back to the Go field name; jsonstrict uses the
  tag verbatim (unmarshal.go:119-122). Minor divergence from the
  "matching encoding/json behavior" claim in the `knownJSONKeys` comment.
  Fixed with an `isValidTag` mirroring stdlib.
  (`TestUnmarshal_InvalidTagNameFallsBack`)

## Recommendations

- [ ] **Nested checking (biggest feature)** — diagnostics stop at the top
  level; unknown/missing fields inside nested objects go unnoticed. Recurse
  into known struct-typed fields and report dotted paths (`address.zip`
  missing, `address.zipp` unknown).
  - [ ] At minimum, document the current top-level-only scope — it is not
    stated anywhere today.

- [ ] **Cache `knownJSONKeys` per type** — reflection runs on every call; a
  `sync.Map` keyed by `reflect.Type` (as stdlib does for its field cache)
  makes repeated decodes of the same type nearly free.

- [ ] **Differential fuzzing** — the fuzz test only asserts "no panic." Add
  invariants: decode of `v` equals plain `json.Unmarshal`; `Unknown` keys
  disjoint from known keys; `Missing` ⊆ required keys; unknown raw values
  are valid JSON.

- [ ] **Benchmarks** — README claims ~2× memory/CPU; add
  `BenchmarkUnmarshal` vs. a `json.Unmarshal` baseline to back the number
  and catch regressions in CI.

- [ ] **Error-mode convenience** — consider a small helper (e.g.
  `Result.Err()` returning a joined error when either list is non-empty) so
  hard-strict callers avoid boilerplate, without changing the
  diagnostic-first design.

- [ ] **Document nil vs. empty `Unknown` map** — `Result.Unknown` is nil
  when no unknown fields exist; state this on the `Result` type.
